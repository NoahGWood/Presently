Semantic Web Components

  Semantic web components are used to provide additional context for web browsers, search engines, and screen readers to help them determine how to display or serve content and in what order. These components also play an important role in improving the accessibility of a website by providing additional information for website accessibility software to help those with disabilities access websites. Two examples of how semantic tags can be used are the Wordpress.com and makeuseof.com sites.

Semantic Tags Used

  There are a number of semantic tags available for use in html. In the case of the makeuseof site, the first semantic tag used was the <article> tag which is used to wrap the entire main article of the page. Good on their namesake, makeuseof makes good use of this tag to wrap their blog/news article which is one of the enumerated uses specifically outlined by Mozilla Developer Network. One of the major benefits of this tag is that it tells screen readers which content to ignore, so if you visit the makeuseof site and your browser is set-up to be in screen reader mode, it will only display the content within the article. This helps to remove clutter such as advertisements, and it makes your site more accessible to mobile devices which may need a simplified view. The makeuseof site also uses <section> tags to help delineate the main portions of the article. This allows each section of the page to be rendered under its' own heading in screen readers which helps to preserve the structure of the page.

  Contrasting with the makeuseof site, the wordpress site does not wrap its' main content in an article tag, and for generally good reason. The home page is effectively a landing page with many large images. Each major section of the page is still split up using the <section> tag, however, without the article wrapping, not every browser will be able to render this page with a screen reader. The primary reason for not using the <article> tag to wrap this content is because none of the sections are independent of the page itself.

  In addition to these tags, both sites also make use of the <picture>, <img>, and <source> tags. The <picture> and <source> tags are not strictly necessary, however they allow web developers to specify multiple image sources depending on screen size which allows them to better tailor the image to suit the screen size. The <img> tag is simply used to tell the browser or screen reader to render an image from an external source. This is opposed to an <svg> or <canvas> tag which can be used to draw images directly in the browser.

  While these are all perfectly fine uses of semantic tags for images, the makeuseof site also includes the <figure> tag. In this case, it is probably unnecessary, as the <figcaption> tag is not included to describe the pictures, but it is my suspicion that this markup is automatically generated, and if the author needed to add a caption to the figure, this would be the best way to do it to ensure that the text which belongs to the image does not get separated by a screen reader rearranging content on the page. The wordpress site does not include this, however it does make use of the <video> tag. This along with the <source> tag enables them to embed a video in the webpage to be played in the browser. This is a proper use of the semantic tag and allows for their advertisement to play in browser seamlessly, and a proper fallback image to be displayed was provided if the source failed, however a fallback tag was not provided in the event that a browser did not support the <video> tag.

  Finally, the wordpress site uses a <section> tag in lieu of the <footer> tag. While this is technically valid, it is not the ideal tag to use for this. The makeuseof site however, does use the proper footer tag. Because the makeuseof site uses a semantically appropriate footer tag, when viewing the page with a screen reader, the footer will not be rendered (and neither will the <nav> or <header> for that matter). If, however, you were to view the wordpress site, all the content would be rendered because there is no distinct article, and the footer would be rendered as if it were a section with a header and block text.

Design

  The design of these sites are very different from one another. While the wordpress site makes good use of many colors, the makeuseof site sticks to just a few colors. The makeuseof site uses the Roboto font while the wordpress site uses Noto Sans for the primary text (although the CSS does embed quite a few fonts as well to use in different contexts on the blogging platform).

  When it comes to proximity, alignment, and sizing, the wordpress site is very big and spaced out taking up the majority of the page. While this looks okay in a responsive site (e.g. on a mobile phone), it is a bit clunky and annoying to use on a desktop computer simply because of the amount of scrolling one needs to do to view each element and because of how large the elements themselves are. Perhaps most annoyingly, in spite of the large size, the use of multiple images per section makes them too big to be appealing and too small to read without difficulty.

  In contrast, the makeuseof site has a balanced feel to it. On larger screens, the page splits into an article and aside showing additional content. The sizing, alignment, and proximity is also balanced. Images are large and take up nearly the full width of their container (with some margin). Section headers are thick and bolded while the text is also large but not so large to make it difficult to read. There is also plenty of content on the page to give a user sufficient time to pause between scrolling so as to not seem tedious to use the site.

  When it comes to uniqueness, wordpress certainly wins out, however that is not necessarily a good thing. Diverting from the tried and tested desktop layout for large screens may save a bit of time during development, but the primary target of wordpress is those who will use the software to host their own sites, and many of the functions one needs to do with their CMS involve using a desktop PC to accomplish it.

Site Performance Comparison

  In terms of performance, the clear winner is makeuseof. Their site was able to fully load and style the page from cached data in under 19ms compared to wordpress.com which took over 99ms to load from cached data!

  Page loading times could be greatly improved in wordpress by reducing the amount of CSS bloat in their system. The website is a CMS, and so provides a wealth of customizability for the site manager and bloggers who use it, but that comes at the cost of increased page load time due to having to process all the style rules that may be overwritten by later rules. Another improvement might be to turn many of the elements into static images since the markup does not actually do much for this site. Some sections included multiple images which could have been replaced by a single image made by the three reducing the graphics calls needed to be made.

  The makeuseof site could be improved by fixing the errors in the javascript the site loads. Currently, the scripts take up 53% of the page load time, and they present a plethora of warnings, errors, and extraneous logs. The most time consuming is a script which attempts to draw a few colors on the screen which do not actually show on the screen, this is likely an extraneous function that was used for testing and never removed.

Accessibility Comparison

  Perhaps somewhat surprisingly, wordpress actually scored better in accessibility testing peformed by webaccessibility.com receiving a score of 75% compared to makeuseof's 72%

  These results are especially surprising given that makeuseof's site works perfectly and expected with the firefox default screen reader while wordpress does not, however, upon deeper analysis, it becomes clear that there are hundreds of issues with the makeuseof site. Most of these issues relate to keyboard compatibility followed by text labels and finally contrast. With wordpress, nearly all of issues relate to contrast, a few to text labels, and only two to keyboard issues.

Summary

  In conclusion, while the overall site design of makeuseof.com is much better than the wordpress.com landing page, makeuseof has some serious accessibility issues to account for. Likewise, wordpress may want to consider making their site a bit more screen reader friendly. While both sites are generally performant, wordpress is the clear loser in regard to performance and should seriously consider removing a lot of the bloat and reducing the number of external images they call as well.
